# Protocol Layer

The project uses two coexisting protocols: gRPC (main wasm evidence flow) and Hydra TCP (zero-knowledge identity subsystem). They are independent and share no connection.

- gRPC contract lives in `protos/attestation.proto`, compiled by `tonic-build` and re-exported by the `protos` crate
- Hydra TCP framing and message tags are defined by `hydra-toolkit`; there is no IDL

## gRPC flow

Two attestation modes are supported, selected via `AttestationMode` in `GetEvidenceRequest`.

### Background-check

```
RP --GetEvidence(mode=BACKGROUND_CHECK, nonce)--> attester --AA/TEE--> evidence
RP <-(evidence, wasm_component)-- attester
RP --Verify(tee_type, nonce, evidence, wasm_component)--> verifier
RP <-(EAR JWT)-- verifier
RP locally verifies EAR signature + compares eat_nonce == local nonce
```

Nonce is generated by the RP (32 random bytes recommended). Replay windows are enforced by the RP holding the nonce ↔ EAR association; the verifier is stateless.

### Passport

```
RP --GetEvidence(mode=PASSPORT, nonce=empty)--> attester
  └─ attester generates nonce → builds evidence → calls verifier gRPC → gets EAR
RP <-(EAR JWT)-- attester
RP locally verifies EAR signature + checks iat freshness (5-minute window)
```

In passport mode the attester generates its own nonce, builds evidence, and calls the verifier internally. The RP receives a ready-to-use EAR without touching evidence or the verifier directly. The attester must have `verifier_endpoint` configured.

Passport is orthogonal to tee_type — `*-hydra` paths still travel background-check or passport for the wasm evidence layer, while hydra zero-knowledge verification happens entirely on the Hydra TCP channel.

## Hydra TCP channel

Three peers stay connected. `hydra-toolkit` defines the frame format:

```
[u64 big-endian length][payload]
```

The first 4 bytes of the payload are a tag (present on every message except the verifier→attester encrypted reply):

| Tag | Meaning | Direction |
|-----|---------|-----------|
| `DEVI` | SignedDeviceClientInfor (attester pubkey + measurement + merkle_leaf + timestamp/period + attester signature) | attester → verifier |
| `PCTX` | plaintext PublicContext (root list + verifier_pk) | verifier → attester / RP |
| `EVID` | EvidenceReply + attester signature over the reply | attester → RP |
| (untagged ciphertext) | ResponseDeviceInfor + PublicContext sealed via ECDH+HKDF-derived KEK, AES-GCM | verifier → attester |

Unknown tags trigger an `error: ...` byte-stream reply on the RP side; attesters treat `error:` / `verification failed:` prefixes as failure and disconnect.

See [hydra.md](hydra.md) for the full data flow, batch semantics, and persistence layout.

## gRPC services

| Service | Method | Caller | Description |
|---|---|---|---|
| `AttesterService` | `GetEvidence` | RP → attester | Push nonce, receive evidence (or EAR in passport mode) |
| `VerifierService` | `Verify` | RP / attester → verifier | Submit evidence, receive EAR |

Full message definitions in `protos/attestation.proto`.

`VerifyRequest.wasm` is a `oneof`, pick one:
- First submission: `wasm_component` (raw wasm bytes)
- Subsequent reuse: `wasm_component_id` (stable ID returned by verifier)

## TeeType enum

| proto value | kebab (used in claims) | notes |
|---|---|---|
| `MOCK = 1` | `mock` | skip real verification |
| `CCA = 2` | `cca` | ARM CCA |
| `CCA_HYDRA = 3` | `cca-hydra` | CCA + hydra channel |
| `TDX = 4` | `tdx` | Intel TDX |
| `TDX_HYDRA = 5` | `tdx-hydra` | TDX + hydra channel |
| `CSV = 6` | `csv` | Hygon CSV |
| `CSV_HYDRA = 7` | `csv-hydra` | CSV + hydra channel |
| `ITRUSTEE = 8` | `itrustee` | iTrustee |
| `VIRTCCA = 9` | `virtcca` | VirtCCA |
| `ITRUSTEE_HYDRA = 10` | `itrustee-hydra` | iTrustee + hydra channel |
| `VIRTCCA_HYDRA = 11` | `virtcca-hydra` | VirtCCA + hydra channel |

The `tee_type` in attester config is a kebab string; the `tee_type` in requests is the proto enum number. Attester rejects any request whose `tee_type` differs from its own configuration.

`*-hydra` appraisers behave identically to the non-hydra variants, only their emitted `tee_type` claim carries the `-hydra` suffix; the hydra identity proof itself is handled on the Hydra TCP channel.

## Nonce encoding

| Use | Encoding |
|---|---|
| `GetEvidenceRequest.nonce` / `VerifyRequest.nonce` | raw bytes (proto `bytes`) |
| RP logs / EAR `eat_nonce` | base64url no-pad |
| `nonce` field inside CCA evidence JSON | base64url no-pad |
| `runtime_data` query parameter on AA REST | standard base64 |

## EAR output

The verifier issues an ES256 JWT EAR. Top-level claims:

```text
iss            = "unified-attestation-verifier"
iat            = unix seconds (issuance)
exp            = unix seconds (iat + 3600)
eat_profile    = "tag:github.com,2024:unified-attestation"
eat_nonce      = base64url(RP nonce)
tee_type       = "mock" | "cca" | "cca-hydra" | "csv" | "csv-hydra" | "tdx" | "tdx-hydra" | "itrustee" | "itrustee-hydra" | "virtcca" | "virtcca-hydra"
component_id   = wasm component ID
verifier_id    = { developer: "unified-attestation" }
submods        = claims map returned by wasm (includes per-TEE measurements)
trust_vector   = { instance_identity, configuration, executables }
```

Any third party holding the verifier public key can locally verify + decode + compare `eat_nonce == local nonce`:

```bash
relying-party \
    --attester http://127.0.0.1:9000 \
    --verifier http://127.0.0.1:8080 \
    --tee-type mock \
    --pubkey config/keys/ear_public.pem
```

`executables < 2` is treated as untrusted.

### Trust Vector dynamic assignment

`trust_vector` is not hard-coded; it is assigned based on verification results:

| TEE type | `instance_identity` | `configuration` | `executables` |
|---------|---------------------|-----------------|---------------|
| mock | 2 | 2 | 2 |
| CCA / CCA-Hydra | nonce_bound ? 2 : 0 | secured/secured_no_debug→2, not_secured/recoverable→1, unknown→0 | 2 |
| CSV / CSV-Hydra | nonce_bound ? 2 : 0 | 2 | 2 |
| iTrustee / iTrustee-Hydra / VirtCCA / VirtCCA-Hydra | nonce_bound ? 2 : 0 | 2 | 2 |
| TDX / TDX-Hydra | 2 | 2 | UpToDate→2, SWHardeningNeeded/ConfigurationAndSWHardeningNeeded→1, OutOfDate/Revoked→0, unknown→1 |

AR4SI values: 2 = Affirming, 1 = Warning, 0 = None.
